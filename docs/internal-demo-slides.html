Problem Framing

用户真正缺的不是更多数据， 而是更快形成判断的路径。

目标用户是“今天看新能源，明天看医疗器械”的私募投资者。他们已经有 Wind / iFinD，但仍然需要在几小时内建立陌生行业的 80 分认知。

“我不需要你替我研究透，我需要你快速帮我排除 95% 的噪音。”
核心诉求：让精力集中到最有价值的下一步动作上，而不是被信息洪水拖慢。
Pain Signals
  • 01
    行业切换极快 需要跨赛道迅速建立上下游、技术路线和风险认知，不能靠慢阅读。
  • 02
    概念噪音过多 小作文、热点蹭概念、产业链误读频繁出现，可信度验证必须前置。
  • 03
    工具分散而不连贯 数据、研究、执行建议分散在不同工具里，缺少从问题到动作的一条链。
  • 04
    真正稀缺的是时间 决策窗口很短，平台必须把“排除无效标的”的速度做得足够快。
Product Thesis

这不是另一个“股票列表页”， 而是一套投研工作流操作系统。

项目目标不是替代全能终端，而是成为私募投资者的效率加速器，用自然语言、 多 Agent 编排和证据验证系统把动作链路缩短。

Not a Terminal

不做全能数据终端

不复制 Wind / iFinD 的全量功能，而是把“下一步该看什么”收束得更明确。

Natural Language

自然语言驱动筛选

用户说“ROE 连续增长且高负债剔除”，系统落到 `FilterGroup` 与 `ScoringConfig`。

LangGraph Runtime

多 Agent 不是黑盒

每个节点都有状态、进度和事件流，用户知道现在卡在哪、结果怎么来的。

Evidence First

先验证，再兴奋

概念可信度拆成主营匹配、订单专利、历史蹭热点和产业链逻辑四个维度。

End-to-End Loop

输出不是一篇更长的报告， 而是一次可执行的下一步动作。

平台围绕投资者当天的判断链路展开：先把问题结构化，再缩小候选池，最后给出研究结果与组合动作。

01 Define

描述问题

从行业、题材或财务条件出发，先把今天要解决的问题说清楚。

02 Screen

缩小候选池

让筛选策略把广市场快速压缩成可研究的机会池和评分列表。

03 Research

行业快研

用标准 5 节点工作流产出行业概览、热度判断、候选股和风险催化剂。

04 Verify

公司深挖与验证

对少量核心标的做概念可信度和证据强度校验，避免被热点带节奏。

05 Act

给出组合动作

将推荐动作、风险预算和建议仓位放回投资决策工作台里完成闭环。

Output 候选池 Output 研究报告 Output 可信度评分 Output 组合建议

整个平台围绕“下一步要看什么、为什么看、看完以后做什么”来组织，而不是把更多信息堆到用户面前。

LangGraph Orchestration

`quick_industry_research` 已经是一个可讲故事的标准模板。

当前仓库中已落地的标准工作流包含 5 个顺序节点，把“快速了解一个陌生行业”的动作拆成可观察、可扩展的链路。

Agent 01

行业概览

先建立产业链、技术路线和市场规模的共同上下文。

Agent 02

市场热度分析

判断题材热度、催化剂与资金关注方向，不让情绪信息失控。

Agent 03

候选股筛选

用数据与规则缩出 5 到 10 个值得继续看的标的。

Agent 04

可信度评估

对热点概念逐一做证据检查和风险提示，避免“概念即逻辑”。

Agent 05

竞争格局总结

形成 Top Picks、竞争位置和最终研究摘要。

Runtime

可回放的运行明细

`WorkflowRun`、`WorkflowNodeRun` 与 `WorkflowEvent` 让过程和结果都能被查看。

What it Produces

30 分钟内拿到一份能继续判断的行业快研包

01 行业概览 02 热度结论 03 候选股 04 风险标签

和纯问答助手的差异在于：任务被拆成有状态的节点，平台知道自己现在执行到哪一步，用户也能看到。

Why it Matters
  • A
    适合做模板化沉淀 工作流配置存数据库后，后续可以由用户自定义、复制和微调。
  • B
    适合做运行时治理 取消任务、查看进度、回看节点事件这些能力都自然成立。
  • C
    适合向外扩展 后续新增公司研究、择时等链路时，结构上已经准备好了。
Architecture Snapshot

技术选型不是堆叠名词， 而是在为“快而可控”的工作流让路。

当前实现遵循 T3 Stack + Python FastAPI 微服务的混合架构，DDD 分层负责边界清晰，异步 Worker 负责长任务解耦。

Surface

Next.js 15 + App Router

负责 UI、页面编排和登录态，首页已汇总研究、筛选和组合建议。

Application

tRPC 作为应用层入口

Router 只做输入校验、编排和错误映射，业务逻辑不直接堆在接口层。

Domain

DDD 限界上下文

`screening`、`workflow`、`intelligence` 等边界上下文把规则、聚合和领域服务分开。

Persistence

Prisma + PostgreSQL

已覆盖 `ScreeningStrategy`、`ScreeningSession`、`WorkflowRun` 等核心模型。

Runtime

Redis + 双 Worker

`screening-worker` 与 `workflow-worker` 分别处理筛选任务和长时工作流执行。

Data Gateway

FastAPI + AkShare

外部金融数据被收敛到基础设施层 HTTP Client，领域层不直接依赖第三方服务。

Rule 领域层保持纯净 Rule 应用层无状态 Rule 数据源可替换
