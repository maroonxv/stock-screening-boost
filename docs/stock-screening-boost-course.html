01
Product Journey

先看用户一天里真正会点什么

这不是一个“股票列表页”。它像一张投研作战桌：先筛选，再研究，再给动作建议，最后把结果沉淀成下一轮经验。

这个产品在帮谁省步骤？

如果你是用自然语言驱动 AI 写软件的人，最重要的第一步不是记技术名词，而是先知道产品在帮用户做哪几段决策压缩。

筛选

把噪音压缩成候选池

先收窄范围，再决定值不值得研究。

研究

把主题问题变成流程

不是一段 prompt，而是可追踪的工作流。

择时

把结论压成动作建议

最后落到观察、试仓、加仓、减仓这些动作。

复盘

把经验喂回系统

建议和结果都会留下来，下一轮不是从零开始。

为什么这对你指挥 AI 很关键？

因为以后你不是说“改首页”，而是要知道新增能力属于筛选、研究、择时还是复盘这条链。

04
Screening Workbench

筛选工作台像实验台：只在你按下“获取”时真正启动仪器

用户先在本地搭好实验条件，只有在需要真实数据时才触发后端和 Python 服务。这让系统更可控，也更省外部请求。

筛选链路的四个关键动作

Action 1

本地搜股票

先在本地映射里搜，不必每输入一个字都打远程服务。

Action 2

选指标和公式

公式先经过后端校验和规范化，避免坏表达式混进去。

Action 3

点击获取才拉数据

这一步才真正通过 tRPC 到 Python 数据服务。

Action 4

结果回本地再过滤

排序和规则过滤尽量在浏览器里做，交互更快。

后端为什么先做“指标合法性检查”？

把它想成安检：TRPCError 不负责飞行，但会先把明显危险的东西拦下来。

code
const client = new PythonScreeningWorkbenchClient();
const catalog = await client.listIndicatorCatalog();
const catalogMap = new Map(catalog.items.map((item) => [item.id, item]));

const indicators = input.indicatorIds.map((indicatorId) => {
  const indicator = catalogMap.get(indicatorId);
  if (!indicator) {
    throw new TRPCError({
      code: "BAD_REQUEST",
      message: `未知指标: ${indicatorId}`,
    });
  }
  return indicator;
});

05
Async Workflow

研究流程不是“点一下等结果”，而是后台工人持续接单

没有 worker、LangGraph 和事件流，研究页就只能卡在一个同步请求里。真正可用的系统需要会排队、会恢复、会汇报进度。

异步研究的真实节奏

1
前端先创建 WorkflowRun用户点按钮，服务端先写一条“准备执行”的运行记录。
2
后台 worker 轮询待处理任务它会先看有没有可恢复的 running run，再看有没有新的 pending run。
3
每个节点都写进进度和事件所以用户在详情页能看到当前跑到哪一步。

worker 的循环像值班经理

code
async function main() {
  console.info(`[workflow-worker] started: ${workerId}`);

  while (!shuttingDown) {
    try {
      const recovered = await executionService.executeRecoverableRunningRun(workerId);

      if (recovered) {
        continue;
      }

      const picked = await executionService.executeNextPendingRun(workerId);

      if (!picked) {
        await sleep(pollIntervalMs);
      }
    } catch (error) {
      const message = error instanceof Error ? error.message : "unknown error";
      console.error(`[workflow-worker] loop error: ${message}`);
      await sleep(pollIntervalMs);

06
Python Gateway

Python 微服务像外勤站，专门负责接金融数据的脏活累活

主站是 TypeScript，但金融数据接入放在 Python 侧。这不是为了炫技，而是为了隔离最可能变、最依赖第三方生态的边界。

为什么单独拆一个 Python 运行时？

Isolation

第三方依赖单独关起来

金融数据生态、Python 包和网络异常都集中在一个可替换边界里。

Stability

让 TypeScript 业务层更干净

主站不必知道 AkShare 的细节，只知道自己在调用一个 HTTP 数据网关。

Replaceability

以后换 provider 影响面更小

AkShare、iFinD 之类的变化，优先留在 Python 侧消化。

routers/ HTTP 入口
gateway/ 协议和错误包装
services/ 数据整理和分析逻辑
providers/ 最靠近外部数据源的一层

一个好网关，不只是转发数据，还要翻译错误

把它想成海关窗口：FastAPI 不只是开门，还要决定错误用什么格式说清楚。

code
@app.exception_handler(GatewayError)
async def handle_gateway_error(request: Request, exc: GatewayError):
    request_id = getattr(request.state, "request_id", create_request_id())
    payload = GatewayErrorResponse(
        meta=build_meta(
            request_id=request_id,
            provider=exc.provider,
            started_at=0,
            cache_hit=False,
            is_stale=False,
            warnings=exc.warnings,
        ),
        error=GatewayErrorBody(code=exc.code, message=exc.message),
    )
    return JSONResponse(status_code=exc.status_code, content=payload.model_dump(mode="json"))

07
Data Loop

数据库不是仓库尽头，而是整个决策闭环的记忆层

最后一站，不再只看“跑起来”。我们看系统如何记住工作台、schema、运行、建议和复盘，让下一轮判断站在上一轮的肩膀上。

五个最值得记住的数据主角

1

ScreeningWorkspace

保存筛选配方，而不是一次性结果。

2

WorkflowRun

记录研究流程的输入、状态、进度和输出。

3

TimingAnalysisCard

保存单股或批量信号的判断卡片。

4

TimingRecommendation

真正面向动作的建议，带着预算和理由。

5

TimingReviewRecord

事后核对建议表现，再反喂给下一轮。
